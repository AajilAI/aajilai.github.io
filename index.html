AIDX · Part 1

Persistent Memory

Cross-session AI memory — every feedback correction, project decision, and user preference persists beyond context window resets. Powered by the gastownhall/beads pattern.

4 Memory Types
TypeStores
userRole, expertise, preferences — calibrates agent tone and depth across sessions
feedbackCorrections + confirmed approaches — never repeat the same mistake twice
projectDecisions, context, blockers — why things are the way they are
referencePointers to external systems — where to look, not what's there
Example memory file
meta/memory/feedback_ci_variables.md
In CI reviews, never use YAML-level variable blocks — always recommend GitLab CI/CD variables. Why: prior incident where YAML vars leaked into job logs.
Architecture
  • ▸meta/memory/ — one markdown file per memory, version-controlled alongside the platform brain. Each file has frontmatter type, name, and description.
  • ▸MEMORY.md index — a pointer-only index file loaded into every Claude Code session automatically. Max 200 lines — every entry is a one-liner.
  • ▸sync-memory-index hook — regenerates MEMORY.md on every memory file write. Zero manual maintenance.
  • ▸gastownhall/beads — open-source pattern for file-per-memory with structured frontmatter. Enables cross-LLM, cross-session context continuity.
Memory survives context window resets, model upgrades, and new conversations. An engineer's preferences from month one are still respected on month twelve.

Architecture

Plan B for simple services, Plan C for services needing runtime .env construction. Both approaches use GitHub as the CI gateway — no long-lived credentials in workflows.

03
Secrets Architecture · Part 3

Plan B vs Plan C

Plan B · GitHub Org Secrets
  • ▸How: Secrets stored as GitHub org-level secrets/vars, injected directly into the Actions environment
  • ▸Naming: SERVICE_PREFIX_ENV_ORIGINAL_KEY
    e.g. BBGITHUB_DEV_DATABASE_URL
  • ▸Use when: Service has <50 secrets, no complex .env construction needed, simple key→env injection
  • ▸Service: asquare (only remaining Plan B service)
Plan C · GCP Secret Manager + WIF
  • ▸How: Secrets stored in GCP Secret Manager. CI authenticates via Workload Identity Federation — zero long-lived credentials
  • ▸Naming: SERVICE_PREFIX_ORIGINAL_KEY
    e.g. ASQUARE_DATABASE_URL
  • ▸Use when: Service needs full .env file construction, large secret count, or multi-environment support
  • ▸Services: bookish-beaver, rsquare, wsquare, tsquare, csquare (all except asquare)
Sensitivity classification — All secrets tagged as SECRET (token, key, password, private key) or VAR (URL, host, port, config) at push time. Only SECRET-class values become masked GitHub secrets; VAR-class become unmasked variables.

Critical Gap · AIDX

Missing: EVAL Harness

We ship standards, agents, and hooks — but have no systematic way to know if they're working. The platform has no quality floor.

Why This Is a Weak Point
  • !Model upgrades are blind. When Claude 4.7 ships, we have no benchmark to know if our agents regress. We find out in production — or not at all.
  • !Hook changes are untested. A broken hook silently degrades every session. No test suite catches it before it ships.
  • !Agent quality is purely manual. Output correctness depends on Maysra catching issues in review. That doesn't scale to 10+ engineers using AIDX.
  • !Standards drift is invisible. We can't tell if agents are actually following meta/lex/ rules or just appearing to. No coverage metric exists.
What the Harness Should Do
  • ▸Golden dataset. Representative prompts per agent — each with known-good outputs and machine-checkable assertions (format, presence of required sections, no banned patterns).
  • ▸Regression CI job. Runs on every AIDX change (hooks, agents, lex). Fails the build if quality score drops below threshold.
  • ▸Model upgrade gate. Before adopting a new Claude version, run the full suite. Promote only if pass rate ≥ baseline.
  • ▸LLM-as-judge scoring. Use Claude itself to score outputs against rubrics defined in meta/lex/ — structured verdict, not free-form commentary.
Without this, AIDX is a bet that things are working. With it, quality becomes measurable and the platform is safe to scale to the full team.
